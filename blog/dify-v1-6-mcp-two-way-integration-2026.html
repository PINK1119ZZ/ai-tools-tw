MCP 是什麼？為什麼 Dify 加入它很重要？

MCP（Model Context Protocol）是 Anthropic 於 2024 年底推出的開放標準，讓 AI 助理能夠標準化地「發現工具、呼叫工具、回傳結果」。你可以把它想成 AI 版本的 USB-C：只要工具支援 MCP，所有 AI 助理都能直接插上去用，不需要為每個工具都寫特製的 plugin 或 API wrapper。

在 Dify v1.6.0 之前，如果你要讓 Claude Code 呼叫 Dify 的 Workflow，你需要：

  1. 把 Dify Workflow 包成 REST API
  2. 自己寫 OpenAPI 文件
  3. 在 Claude Code 端自定義工具格式
  4. 手動處理認證、錯誤格式、超時邏輯

v1.6.0 之後，這一切壓縮成一個 URL。

💡 核心概念

Dify v1.6.0 雙向 MCP = Dify 作為 MCP Client（呼叫外部工具）＋ Dify 作為 MCP Server（被外部 AI 呼叫）。這個對稱架構讓 Dify 成為 MCP 生態系的真正中樞。

在 Dify 的 Tools 頁面，你現在可以直接新增任何 MCP Server 作為工具。以下是目前最實用的幾個整合：

📌 5 分鐘設定：在 Dify Agent 裡加入 Zapier MCP

Step 1 — 取得 Zapier MCP URL

前往 mcp.zapier.com → 登入 → 複製你的個人化 MCP URL（格式如 https://mcp.zapier.com/api/mcp/v1/{your-token}/sse）


Step 2 — 在 Dify 新增 MCP Server

工作室 → Tools → MCP → 新增 MCP Server
・名稱：Zapier
・URL：貼上 Step 1 的 URL
・Server ID：zapier（自訂英數）


Step 3 — 在 Agent 裡啟用工具

建立/編輯 Agent → 工具 → 選擇 Zapier → 選擇所需工具（Gmail、Slack 等）


Step 4 — 撰寫 System Prompt

「你是一個業務助理，可以使用 Zapier 工具傳送 Gmail、更新 Google Sheets、發送 Slack 通知。當使用者提出需求時，主動選擇合適的工具執行。」

方向二：把 Dify Workflow 發佈為 MCP Server

這是 v1.6.0 最顛覆性的功能。任何你在 Dify 建立的 Workflow 或 Agent，現在都可以一鍵發佈為標準 MCP Server 端點。換句話說：你用視覺化拖拉積木建的業務邏輯，Claude Code 可以直接呼叫它。

發佈步驟（2 分鐘完成）

在 Dify 端

發佈 → 訪問 API → 開啟「MCP Server」→ 複製 MCP URL
格式：https://api.dify.ai/v1/mcp/{app-id}/sse


在 Claude Code 端（~/.claude/settings.json）

{
  "mcpServers": {
    "dify-crm-agent": {
      "url": "https://api.dify.ai/v1/mcp/{your-app-id}/sse",
      "headers": {
        "Authorization": "Bearer {your-api-key}"
      }
    }
  }
}


在 Cursor 端（settings.json）

"mcp": {
  "servers": {
    "dify-crm-agent": {
      "url": "https://api.dify.ai/v1/mcp/{your-app-id}/sse",
      "apiKey": "{your-api-key}"
    }
  }
}

⚠️ 重要限制（截至 2026-04-25）
  • 目前僅支援 HTTP/SSE 傳輸，尚不支援 stdio
  • 自架版需確保 Dify 對外有公開 HTTPS URL
  • 每個 Dify App 只能暴露一組 MCP 端點（一個 Workflow = 一個 MCP Server）
  • Streaming tool call 功能仍在開發中

常見問題 Q&A

❓ Dify v1.6.0 的雙向 MCP 是什麼意思？

雙向 MCP 指 Dify 既能作為 MCP Client（呼叫外部 MCP Server，例如 Zapier、Linear、Notion），也能作為 MCP Server（把自己的 Workflow 或 Agent 發佈成標準 MCP 端點，供 Claude Code、Cursor、n8n 等工具呼叫）。

❓ Dify MCP Server 和一般 REST API 有什麼不同？

REST API 需要手動撰寫文件、設定認證、處理錯誤格式。MCP Server 是標準化協議，Claude/Cursor 等 MCP Client 可以自動發現工具能力（tool discovery）、理解參數、處理錯誤回傳，無需額外文件，大幅降低整合成本。

❓ Dify v1.6.0 的 MCP 支援哪些傳輸協議？

目前僅支援 HTTP-based MCP（SSE），遵循 MCP Protocol version 2025-03-26。stdio 模式尚未支援。建議使用 HTTPS 端點確保安全性。

❓ 自架 Dify 可以使用雙向 MCP 嗎？

可以。升級至 Dify v1.6.0 後（GitHub 最新版），自架版同樣享有完整雙向 MCP 功能，無需額外付費。Dify Cloud 版本的付費方案也已包含此功能。

❓ Zapier MCP 整合要額外付費嗎？

Zapier 本身的費用與以往相同（免費版有任務數限制，付費版從 $20/月起）。Dify 端不收取額外費用。Zapier 的 MCP 端點是免費提供的，費用來自實際執行的 Zap 任務數量。
